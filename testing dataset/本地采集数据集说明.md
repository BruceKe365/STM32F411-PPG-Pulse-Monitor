# 本地 MAX30102 采集与回放数据说明

更新时间：2026-06-24

本目录保存本项目使用 STM32F411 + MAX30102 实际采集的 PPG 数据，以及使用 PC 脚本对这些数据进行离线回放后生成的结果。这里主要用于验证：

- 原始红光、红外 PPG 数据是否连续；
- 手指检测、心率、血氧是否稳定；
- 可靠峰值和 PPI 是否存在假峰、漏峰；
- 0.91 寸 OLED 波形显示是否正常；
- AF 风险和压力指数的实机输入质量与稳定策略。

本目录与 `training_dataset/` 的职责不同：

| 目录 | 内容 |
| --- | --- |
| `testing dataset/` | 本机 MAX30102 实际采集、PC 离线回放、硬件和算法边界测试。 |
| `training_dataset/` | PhysioNet、WESAD 等公开数据集，模型参数和全量验证报告。 |

> 本目录没有真实 AF 患者 PPG，也没有经过实验标签确认的心理压力训练数据。这里的数据只能作为本地正常/边界烟测，不能当作 AF 阳性或 Stress 标签训练集。

## 每个采集目录中常见的文件

不同阶段的文件名略有差异，例如早期可能使用 `processed.csv`，后期可能使用 `raw_processed.csv`。判断文件类型时以内容和后缀为主。

| 文件形式 | 内容 | 推荐用途 |
| --- | --- | --- |
| `raw.csv`、`*_raw.csv` | STM32 USB CDC 输出的原始红光/红外样本。 | 检查丢样、原始波形、传感器光强。 |
| `processed.csv`、`*_processed.csv` | PC 算法逐样本处理结果。 | 最常用；查看滤波波形、质量、HR、SpO2、峰值和 PPI。 |
| `events.csv`、`*_events.csv` | 人工事件标记。 | 对齐“放手指”“移开手指”等实验动作。 |
| `summary.json`、`*_summary.json` | 一次采集或回放的汇总统计。 | 首先阅读；快速判断时长、样本数、丢样和最终结果。 |
| `replay_*`、`*_replay_*` | 使用不同参数对已有 raw 数据离线回放的结果。 | 比较算法版本，不代表重新采集。 |
| `frames/` | OLED 波形预览帧和列推进记录。 | 检查小屏波形样式和峰值标记。 |

## 推荐查看顺序

面对一个采集目录时，建议按以下顺序阅读：

1. 打开 `summary.json`，确认采集时长、样本数、参数和丢样情况。
2. 打开最新的 `*_processed.csv`，查看 `finger_present`、`finger_score`、`hr_bpm`、`spo2`。
3. 检查 `peak` 和 `peak_interval_ms`，判断是否存在假峰或漏峰。
4. 需要查看传感器原始状态时，再打开 `raw.csv` 的 `red`、`ir`。
5. 如果同目录存在多个 `replay_*`，根据文件名确认使用的参数，不要混合比较。
6. 最终 AF/压力结论回到 `training_dataset/reports/` 的全量报告查看。

## 如何看 `summary.json`

以 `112_subject_A_pressure_baseline_light_stable_20260617_140429/raw_summary.json` 为例，主要字段如下：

| JSON 字段 | 含义 | 正常判断 |
| --- | --- | --- |
| `elapsed_s` | 实际采集时长。 | 90 s 采集应接近 90。 |
| `samples` | 样本总数。 | 100 Hz、90 s 应约为 9000。 |
| `parameters` | 本次 PC 处理使用的阈值和滤波参数。 | 比较 replay 时先确认参数是否一致。 |
| `data_quality.missing_sample_count` | 根据 `sample_count` 检测的丢样数。 | 理想值为 0。 |
| `data_quality.nonmonotonic_sample_count` | 样本序号是否倒退。 | 理想值为 0。 |
| `data_quality.sample_count_gap` | 相邻样本序号差。 | 正常应始终为 1。 |
| `raw_stats` | 红光、红外原始值统计。 | 用于判断光强和接触变化。 |
| `processed_stats` | RMS、质量分、HR、SpO2 等统计。 | 用于快速判断整段稳定性。 |
| `final` 或最终状态字段 | 结束时的手指、HR、SpO2 状态。 | 移开手指后显示为空是正常行为。 |

不要只看 JSON 中的最大值。瞬时最大值可能来自接触变化或伪迹，应结合中位数、标准差和 processed CSV 的时间序列判断。

## 如何看原始 `raw.csv`

后期标准 raw 表头：

```text
session_id,label,pc_time,t_ms,red,ir,irq_count,sample_count
```

| 列 | 含义 |
| --- | --- |
| `session_id` | 本次采集或回放的会话标识。 |
| `label` | 人工设置的场景标签。 |
| `pc_time` | PC 收到该行数据的时间。 |
| `t_ms` | MCU 运行时间戳，单位 ms。USB 缓冲会使其间隔略有波动。 |
| `red` | MAX30102 红光原始 ADC 值。 |
| `ir` | MAX30102 红外原始 ADC 值，是心率和峰值分析的主要输入。 |
| `irq_count` | 传感器中断累计次数。 |
| `sample_count` | MCU 样本累计序号，是判断是否丢样的首选字段。 |

### 原始数据首先检查什么

1. `sample_count` 应严格递增，相邻差通常为 1。
2. 100 Hz 采样时，90 s 数据应约有 9000 行。
3. `red`、`ir` 放上手指后应明显高于无手指状态，并存在周期性小幅波动。
4. 判断真实采样连续性时优先使用 `sample_count`，不要只看 `pc_time` 或 `t_ms`。
5. USB 一次可能批量发送多行，因此 `pc_time` 或 `t_ms` 的瞬时间隔不完全等于 10 ms，不一定表示丢样。

## 如何看处理后的 `processed.csv`

较新的 processed CSV 表头：

```text
session_id,label,pc_time,t_ms,sample_time_ms,sample_count,
red,ir,fs_hz,red_dc,ir_dc,red_ac,ir_ac,red_filt,ir_filt,
red_rms,ir_rms,finger_score,finger_present,hr_confirmed,
hr_bpm,hr_peak_bpm,hr_autocorr_bpm,spo2,ratio,peak,peak_interval_ms
```

旧文件可能缺少 `sample_time_ms`、`hr_confirmed` 或部分诊断列，这是历史格式差异，不代表文件损坏。

### 信号处理列

| 列 | 含义 | 怎么看 |
| --- | --- | --- |
| `sample_time_ms` | 用 `sample_count × 10 ms` 得到的等间隔时间。 | 画图横轴优先使用它。 |
| `fs_hz` | 估计采样率。 | 稳定阶段应接近 100 Hz。 |
| `red_dc`、`ir_dc` | 红光/红外直流基线。 | 反映手指接触和慢变化。 |
| `red_ac`、`ir_ac` | 去除直流基线后的脉动分量。 | 包含脉搏和噪声。 |
| `red_filt`、`ir_filt` | 进一步平滑后的 PPG。 | 查看波形、峰值时优先使用 `ir_filt`。 |
| `red_rms`、`ir_rms` | 滚动窗口内的交流幅度。 | 过低表示脉动弱，突然升高可能是运动伪迹。 |

### 手指和质量列

| 列 | 含义 | 怎么看 |
| --- | --- | --- |
| `finger_present` | 是否检测到手指，0/1。 | 分析有效测量时先筛选为 1。 |
| `finger_score` | 综合 DC 和 AC 幅度的质量分。 | SpO2 通常要求不低于 4；越大不代表无限更好。 |
| `hr_confirmed` | 首次心率是否通过多候选确认。 | 为 0 时 HR 空白属于正常预热。 |

### 心率和血氧列

| 列 | 含义 | 怎么看 |
| --- | --- | --- |
| `hr_bpm` | 最终稳定后的显示心率。 | 这是验收时主要查看的 HR。 |
| `hr_autocorr_bpm` | 当前自相关心率候选。 | 可用于分析 HR 为什么更新或不更新。 |
| `hr_peak_bpm` | 由峰间期直接估计的辅助心率。 | 对假峰/漏峰敏感，不是最终显示值。 |
| `spo2` | 质量门控后的血氧结果。 | 预热或质量差时空白是设计行为。 |
| `ratio` | 红光/红外 AC/DC 比值。 | 血氧公式的中间量，不直接当血氧读数。 |

### 可靠峰和 PPI 列

| 列 | 含义 | 怎么看 |
| --- | --- | --- |
| `peak` | 当前样本是否被可靠峰检测器接受，0/1。 | 将 `peak=1` 的点叠加到 `ir_filt` 上检查峰位置。 |
| `peak_interval_ms` | 本次峰与上次可靠峰之间的时间。 | 是 AF 和压力 HRV 的 PPI 输入。 |

PPI 与心率大致满足：

```text
PPI_expected_ms = 60000 / hr_bpm
```

例如：

| 心率 | 期望 PPI |
| ---: | ---: |
| 60 bpm | 约 1000 ms |
| 75 bpm | 约 800 ms |
| 100 bpm | 约 600 ms |
| 120 bpm | 约 500 ms |

如果连续出现约一半的 PPI，常见原因是同一拍内检测到两个峰；如果出现约两倍的 PPI，常见原因是漏掉一拍。AF 偶发假高通常应先检查这里。

## 为什么 CSV 中会有空白值

空白不一定是错误，以下情况会主动不输出结果：

- 刚放上手指，HR 和 SpO2 还在预热；
- 心率首次 3 个自相关候选尚未满足一致性；
- `finger_present=0`；
- `finger_score` 太低；
- 红光/红外比值超出血氧有效范围；
- 可靠峰不足，尚不能形成 PPI；
- HRV 窗口长度不足；
- 异常 PPI 比例超过质量门限。

因此不要把空白直接填成 0。0 bpm、0% SpO2 与“当前没有可信结果”含义完全不同。

## 用 Excel 或表格软件快速查看

### 查看心率与血氧

1. 打开 `*_processed.csv`。
2. 筛选 `finger_present=1`。
3. 观察 `hr_bpm`、`spo2` 是否在预热后出现。
4. 对 `hr_bpm` 和 `spo2` 分别插入折线图。
5. 横轴使用 `sample_time_ms`，再除以 1000 可得到秒。

### 查看 PPG 波形

1. 横轴选 `sample_time_ms`。
2. 纵轴选 `ir_filt`。
3. 只截取 10--20 s 范围，避免 9000 行数据在一张图中挤成色块。
4. 再筛选 `peak=1`，检查峰是否落在每个脉搏主峰附近。

### 查看 PPI 是否异常

1. 筛选 `peak=1`。
2. 只保留 `sample_time_ms`、`hr_bpm`、`peak_interval_ms`。
3. 计算或心算 `60000/hr_bpm`。
4. 对比实际 `peak_interval_ms` 是否频繁接近期望值的一半或两倍。

## 如何看事件和其他 CSV

事件文件的标准表头：

```text
session_id,pc_time,event_s,event_label
```

| 列 | 含义 |
| --- | --- |
| `event_s` | 相对采集开始的事件时间，单位 s。 |
| `event_label` | 人工事件说明，例如放上手指、移开手指或改变按压力度。 |

如果事件文件只有表头，表示本次采集没有人工标记，不是文件损坏。

目录中还可能存在以下非逐样本 CSV：

| 文件形式 | 内容 |
| --- | --- |
| `*_comparison.csv`、`*_ranges.csv` | 不同算法参数或预热时间的汇总对比。 |
| `waveform_columns.csv` | OLED 每一列的坐标、模式、滤波值和峰标记。 |
| `*_summary.csv` | 多个文件或窗口的统计汇总。 |

判断方法很简单：先看第一行表头。包含 `red,ir,sample_count` 的通常是采样数据；包含 `case`、`group`、`mean`、`median` 等字段的通常是实验汇总，不应按 PPG 波形绘制。

## 历史和故障记录如何理解

部分目录保存的是失败实验，不能当作算法有效数据：

| 目录 | 正确解释 |
| --- | --- |
| `15_MCU_C版V3串口烟测失败` | 烧录后 COM7 打开失败，0 样本。 |
| `16_MCU_C修复后串口烟测失败` | COM7 打开卡住，0 样本。 |
| `18_MCU_C对齐后串口烟测失败` | 烧录后 COM7 再次卡住，0 样本。 |
| `95_当前固件烧录串口复测` | 最新固件烧录成功，但未重插 Type-C 时串口失败。 |
| `100_60秒提示放手移手空采集` | 空采集尝试，没有可用 raw 数据。 |

这些目录证明的是 Windows USB CDC 重新枚举问题，不代表 MAX30102 数据格式或算法损坏。烧录/OpenOCD reset 后若 COM7 可见但无法打开，应先重插 Black Pill Type-C。
