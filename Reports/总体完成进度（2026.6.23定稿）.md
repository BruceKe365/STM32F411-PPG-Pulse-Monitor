按键显示顺序：


当前串口宏：

```text
RAW_STREAM_USB_ENABLE = 1
```

#### AF 当前策略

AF 方法：PPG robust peak -> PPI -> HRV/不规则节律特征 -> 离散朴素贝叶斯 -> 0-100% AF risk。

```text

```

当前 live AF 行为：

- 未出值或当前风险 >=20% 时每 10s 尝试更新。
- 当前风险 <20% 后每 30s 尝试更新。
- PPI 异常比例超过 20%、窗口不足、PPI 推算 HR 与当前 HR 差异 >18 bpm 时保持旧值。
- 新风险比旧风险上跳超过 20% 时保持旧值；下降不拦截。
- AF test 只播放内置 AFDB `06995` PPI 段，不使用 live 稳定策略。

#### Stress 当前策略

Stress 方法：PPG robust peak -> PPI -> HRV 特征 -> Logistic Regression -> 1-99 stress index。

```text

STRESS_HRV_FIRST_STEP_MS = 10000
```

#### 当前固件 AF 行为

按键使用 PA0 轮询，不依赖 CubeMX 外部中断。显示模式循环：

1. 默认：大屏显示心率/血氧，小屏显示 Pause/Loading/波形。
2. 按一次：大屏显示实时传感器来源的 `AF %`。
3. 再按一次：大屏显示 `[test]` 和 `AF %`，来源切换为内置 AFDB PPI 测试段。
4. 再按一次：回到心率/血氧显示。

live AF 只在这些条件都满足后才开始收 PPI 并计算：

```text
finger_present != 0
ppg_signal_valid != 0
heart_rate_bpm > 0
spo2_percent > 0
```

当前 MCU 端 AF 参数：

```text
AF_WINDOW_TARGET_MS = 30000
AF_RISK_STEP_MS = 10000
AF_MIN_PPI_COUNT = 20
AF_MAX_PPI_COUNT = 100
AF_LIVE_HR_TOLERANCE_BPM = 18.0
```

也就是说，live 模式需要心率/血氧先稳定，再累计最近约 30 秒 PPI，之后每 10 秒刷新一次风险。若 PPI 窗口推算出的 HR 和当前 HR 差异超过 18 bpm，则显示 `--`。

test 模式内置的是 AFDB `06995` 的一段 AF rhythm PPI/RR 间期，不是写死的 85%。进入 test 后需要按 PPI 间隔播放，窗口满约 30.244 秒后才会计算出约 85% 风险。

#### 当前验证结论

当前模型和 MCU 等价逻辑总体能区分稳定正常和明显 AF-like 不规则：

- 本地正常 PPG：`local_ppg` 有效显示点风险为 0%。
- MCU test 内置 AFDB 06995：约 30.244 秒后风险为 85%。
- 当前 MCU 流式显示报告合并 AFDB/LTAFDB/NSR2DB，按 `risk >= 80%` 看有效显示点：AF 标签 45.146%，normal 标签 0.258%，mixed 标签 7.596%，other 标签 18.981%。
- 同一报告中的原始窗口质量门控基线不是流式显示结果：AF 窗口 95.664%，normal 窗口 0.287%。两种统计不可混用。

质量门控实验已经在 `scripts/validate_mcu_af_live.py --quality-gate-experiment` 里合并，不再需要单独新脚本。当前候选结论：

- 不建议强行缩放或压低 live AF 概率，那会让真实 AF 展示变假。
- 更合理的方向是“质量不可信就显示 `--`”，不是把概率硬改低。
- 候选方案 `outlier_frac_gt18pct`：按全部原始窗口统计，normal 高风险从 0.287% 降到 0.192%，AF 高风险从 95.664% 降到 93.117%。
- 候选方案 `artifact_mild`：normal 高风险为 0.145%，AF 高风险为 92.751%。
- 目前这些质量门控只在电脑端验证过，尚未移植进 MCU。

### I2C 与传感器/OLED 接线

| 外设 | STM32 外设 | SCL | SDA | 7-bit 地址 | 备注 |
| --- | --- | --- | --- | --- | --- |
| 0.96 寸 SSD1306 OLED | I2C1 | `PB6` | `PB7` | `0x3C` | 128x64，供电 3.3V，模块自带上拉 |
| 0.91 寸 SSD1306 OLED | I2C2 | `PB10` | `PB3` | `0x3C` | 128x32，供电 3.3V，模块自带上拉 |
| MAX30102 | I2C3 | `PA8` | `PB4` | `0x57` | 供电 3.3V，模块自带上拉 |

CubeMX/HAL 当前 I2C 配置：

| 外设 | 速率 | 地址模式 | GPIO 模式 |
| --- | --- | --- | --- |
| I2C1 | 400 kHz | 7-bit | AF Open-Drain |
| I2C2 | 400 kHz | 7-bit | AF Open-Drain |
| I2C3 | 100 kHz | 7-bit | AF Open-Drain |

重要速率记忆：

- `I2C1` 是 0.96 寸 128x64 SSD1306 大屏，当前必须保持 `400 kHz`，否则大屏整屏刷新会明显拖慢主循环。
- `I2C2` 是 0.91 寸 128x32 SSD1306 小屏，当前必须保持 `400 kHz`，否则心率波形会出现走一段、顿一下的卡顿。
- `I2C3` 是 MAX30102，当前仍保持 `100 kHz`，不要为了屏幕刷新把传感器总线一起改掉。
- 当前 CMake 文件不保存这些外设速率。实际生效位置是 `Core/Src/i2c.c` 里的 `hi2c1.Init.ClockSpeed`、`hi2c2.Init.ClockSpeed`、`hi2c3.Init.ClockSpeed`。
- `cmake/stm32cubemx/CMakeLists.txt` 只是把 `Core/Src/i2c.c` 加入编译；如果以后 CubeMX 或 STM32Cube 插件重新生成 `i2c.c`，必须重新确认 `I2C1/I2C2=400000`、`I2C3=100000`。

注意：HAL 的 I2C API 使用地址时需要左移一位，例如 MAX30102 `0x57 << 1`，SSD1306 `0x3C << 1`。

### MAX30102 中断引脚

| MAX30102 引脚 | STM32 引脚 | CubeMX 配置 | 说明 |
| --- | --- | --- | --- |
| `INT` | `PB5` | `GPIO_MODE_IT_FALLING` + `GPIO_PULLUP` | FIFO/PPG 数据准备好时传感器拉低，走 EXTI9_5 中断 |

## 当前遗留问题

MAX30102 的心率/血氧算法仍不稳定。

已遇到两类问题：

- 阈值太低时：没有手指也可能被环境反射或噪声误判，OLED 会错误显示心率血氧。
- 阈值太高时：手指放上去后一直不显示心率和血氧。

当前代码已经加了：

- `finger_present`：判断是否可能有手指。
- `ppg_signal_valid`：判断是否有可信 PPG 波形。
- 只有 `finger_present == 1` 且 `ppg_signal_valid == 1` 时才显示心率血氧。

但实际表现仍然需要重新调参，当前用户反馈是：手指放上去一直不显示，说明有效波形门槛可能又调得偏高。

主要脚本：

```text

```

当前关键能力：

当前关键参数：

| 参数 | 当前值 | 说明 |
| --- | ---: | --- |
| `window` | `10s` | 滚动分析窗口 |
| `warmup` | `20s` | 检测到手指后等待滤波稳定 |

| `max_peak_ms` | `1200ms` | 最低支持约 `50 bpm` |

当前 HR 算法：

- 主输出使用 IR 滤波波形自相关估计。
- 使用最近自相关结果的中位数平滑。
- 自相关暂时失败时优先沿用历史中位数，减少峰值法瞬时误跳。
- 合成波形已验证 `60/75/90/150/160/180 bpm` 均可识别。
- 当前 HR 范围不是只适合 `80-100 bpm`；峰值/自相关搜索约覆盖 `50-200 bpm`，最终显示门限为 `45-190 bpm`。

当前 SpO2 算法：

```text
ratio = (red_rms / red_dc) / (ir_rms / ir_dc)
SpO2 = 101.0 - 7.0 * ratio
```

SpO2 输出条件：

- `0.40 <= ratio <= 0.85`
- `finger_score >= 4.0`

强按、低灌注或低质量数据即使能算出 HR，也不显示 SpO2，而是显示 `--`。

- 优质数据回放中，HR 和 SpO2 都是在检测到手指后约 `19.8s` 首次出现，符合 warmup 设计。

## 2026-06-09 状态同步：OLED 卡顿根因与 I2C 速率记忆

用户已在硬件端确认：之前 0.91 寸小屏心率波形“一段一顿”、脉搏峰出现时卡顿的核心原因不是心率/血氧算法本身，而是 OLED 所在 I2C 总线仍按默认 `100 kHz` 工作。代码里后来改了速率，但工程/CMake 侧没有形成单独的参数记忆，容易让后续维护者误判。

当前代码检查结果：

```text
Core/Src/i2c.c
hi2c1.Init.ClockSpeed = 400000;  // I2C1: 0.96 寸 128x64 SSD1306 大屏
hi2c2.Init.ClockSpeed = 400000;  // I2C2: 0.91 寸 128x32 SSD1306 小屏
hi2c3.Init.ClockSpeed = 100000;  // I2C3: MAX30102
```

工程文件关系：

- `CMakeLists.txt` 不保存 I2C 外设速率。
- `cmake/stm32cubemx/CMakeLists.txt` 只是把 `Core/Src/i2c.c` 编译进工程。
- 以后如果通过 CubeMX、STM32Cube 插件或重新生成代码覆盖了 `Core/Src/i2c.c`，必须手动复查上述三处 `ClockSpeed`。

维护原则：

- 小屏/大屏刷新卡顿时，先检查 `I2C1/I2C2` 是否仍为 `400000`，不要先怀疑已经定稿的 HR/SpO2 算法。
- MAX30102 仍走 `I2C3=100000`；不要为了 OLED 刷新把传感器总线一起改掉，除非后续单独做传感器链路验证。
- Type-C USB CDC 原始数据输出模块仍应保持独立，不要为了 OLED 显示优化破坏原始 `red/ir` 数据输出格式。

## 2026-06-11 后续方向：端侧疑似房颤风险筛查

用户提出下一阶段可以参考之前在 DSP 开发板上做手写体识别的思路：PC 端训练朴素贝叶斯概率参数，单片机端只保存参数并执行轻量推理。该思路适合本项目，但应定位为“基于 PPG 的疑似房颤风险筛查 / 不规则心律概率提示”，不要表述成医学诊断。

### 总体原则

- 不改动当前已经阶段性定稿的 HR/SpO2 主算法。
- 新增独立模块，例如 `AF_Screening_*`，只读取现有 PPG 峰值/脉搏间期/信号质量结果。
- PC 端负责数据集读取、特征工程、朴素贝叶斯训练、参数导出。
- STM32 端负责在线特征计算和朴素贝叶斯推理，输出 `af_risk_percent` / `af_valid` / `irregular_rhythm_suspected`。
- 用词必须保守：显示“疑似房颤风险”或“不规则心律提示”，不要显示“诊断房颤”。

### 推荐模型路线

优先采用“PPI 节律特征 + 离散化朴素贝叶斯”：

1. MCU 从现有 IR PPG 峰值检测中记录相邻脉搏峰时间差：

```text
PPI = current_peak_ms - previous_peak_ms
```

2. 在 30-60 秒窗口内提取节律特征：

```text
mean_ppi
std_ppi
rmssd
pnn50
cv_ppi = std_ppi / mean_ppi
max_ppi / min_ppi
short_interval_count
long_interval_count
valid_peak_count
signal_quality_score
autocorr_stability / correlation_score
```

3. PC 端把连续特征离散成若干桶，例如：

```text
RMSSD: 0-20, 20-40, 40-80, 80+ ms
pNN50: 0-5%, 5-15%, 15-30%, 30%+
PPI std: 0-30, 30-60, 60-120, 120+ ms
CV: 0-0.05, 0.05-0.10, 0.10-0.20, 0.20+
```

4. PC 端训练得到参数：

```text
log P(Normal)
log P(AF)
log P(feature_bin | Normal)
log P(feature_bin | AF)
```

5. MCU 端推理只做查表累加：

```text
score_normal = logP_normal + sum(logP_bin_normal)
score_af     = logP_af     + sum(logP_bin_af)
```

然后把两个 score 转换成风险比例或直接比较大小。该方法比小神经网络更容易解释、更容易移植，也更符合 STM32F411 的资源条件。

备选方案是高斯朴素贝叶斯：PC 端导出每类每特征的 `mean`、`var`、`prior`，MCU 端计算 log Gaussian。该方案不需要分桶，但 MCU 端涉及平方和 log/exp，可通过 PC 端预计算 `inv_var`、`log_coeff` 降低运算量。

### 可用公开数据集

优先考虑以下公开数据：

- MIT-BIH Atrial Fibrillation Database：长期 ECG，带 AF 节律标注。适合先从 RR 间期训练“不规则节律”模型。
- PhysioNet/CinC Challenge 2017：短单导联 ECG，标签包括 Normal、AF、Other、Noisy。适合短窗口 AF 分类验证。
- MIMIC waveform 类数据库：可能包含 ECG/PPG 等多波形，但标签整理成本更高。
- BIDMC PPG and Respiration Dataset：有 PPG/ECG/呼吸等信号，但不是专门 AF 标签数据，更适合验证 PPG 预处理和峰值检测。

注意：很多公开 AF 数据是 ECG，不是 PPG。可以先用 ECG 的 RR 间期训练节律不规则模型，再迁移到 PPG 的 PPI 间期上，但必须承认存在 ECG->PPG 的 domain gap。更严谨的后续验证是用带 ECG 标签的 PPG 数据，或自行采集 PPG 并用手表/心电参考设备做对照。
